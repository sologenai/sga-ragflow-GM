# 2026-04-04 TK-002 Work Order

## 基本信息

- 工单编号：`TK-002`

- 指派对象：`agent-001`
- 指派人：`agent-x`
- 当前状态：`待验收`
- 优先级：`中`

## 背景

当前管理后台的模型管理页中，一个 provider 下可能挂载多个模型。

现状问题是：

- 页面支持删除整个 provider
- 页面不支持删除 provider 下的单个模型

经项目经理评估，当前问题不是后端能力缺失：

- 后端已经存在单模型删除接口 `/llm/delete_llm`
- 前端 service 已经暴露 `delete_llm`
- 前端 hook 已经存在单模型删除封装

因此，本工单的核心目标不是补后端，而是补齐管理页上的单模型删除入口。

## 目标

在模型管理页中，为 provider 下的每个模型增加“删除”按钮，使管理员可以删除单个模型，而不是只能删除整个 provider。

## 强制实现原则

- 优先采用最小改动方案
- 优先只改前端页面
- 必须复用现有单模型删除接口与 hook
- 不改数据库结构
- 不改后端接口设计
- 不通过环境变量或配置开关回避本问题
- 不影响现有 provider 删除能力

## 建议实现方式

### 1. 只改模型管理页相关前端

目标区域：

- `web/src/pages/user-setting/setting-model/components/modal-card.tsx`
- `web/src/pages/user-setting/setting-model/hooks.tsx`

必要时可调整：

- `web/src/locales/` 下相关文案

### 2. 在模型行补齐删除入口

当前每个模型行已有：

- 本地模型编辑按钮
- 启停开关

本工单要求新增：

- 单模型删除按钮

要求：

- 删除按钮必须作用于当前模型，不可误删整个 provider
- 交互位置要与当前编辑按钮、开关保持协调
- 若存在布局挤压，优先做轻量调整，不要大改 UI 结构

### 3. 复用现有删除能力

要求直接复用现有能力：

- 前端 hook：单模型删除逻辑
- 后端接口：`delete_llm`

不要重复封装一套新接口，不要新增平行实现。

### 4. 增加确认交互

删除单个模型时，必须弹出确认框。

要求确认信息中清楚展示：

- provider 名称
- 模型名称

目标是避免管理员误把“删单模型”和“删整个 provider”混淆。

### 5. 保持现有能力不回归

必须确保以下能力不受影响：

- provider 级删除
- 模型启停开关
- 本地模型编辑能力
- 列表刷新与状态同步
