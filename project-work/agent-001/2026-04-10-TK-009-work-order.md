# 2026-04-10 TK-009 Work Order

## Basic Info

- Ticket: `TK-009`
- Title: `Model management add/delete restricted to superuser with clear ordinary-user guidance`
- Assignee: `agent-001`
- Dispatcher: `agent-x`
- Current Status: `accepted`
- Priority: `high`

## Background

The current model-management page has a permission and UX mismatch:

- ordinary users can see add/delete actions
- ordinary users can click delete
- the UI reports delete success
- the model is not actually deleted

The product requirement is now explicit:

1. only superusers can add models
2. only superusers can delete a model or provider
3. when an ordinary user attempts add/delete, the system must give a clear guidance message such as:
   - `需要新增模型，请联系超级管理员`
   - `需要删除模型，请联系超级管理员`

This ticket must fix both the real permission boundary and the misleading frontend feedback.

## Goal

Deliver a coherent add/delete permission model for model management:

- superuser can add/delete normally
- ordinary user cannot add/delete
- ordinary user receives an explicit guidance message
- frontend must not report fake success on a no-op delete

## Mandatory Implementation Rules

- This is a focused corrective ticket, not a model-management redesign.
- Add and delete must be fixed together.
- The backend must become the source of truth for permission enforcement.
- The frontend must reflect that permission model clearly.
- Do not leave success behavior dependent on a backend no-op.

## Required Scope

### 1. Backend permission hardening

Relevant file:

- `api/apps/llm_app.py`

Required behavior:

- only superusers can call add-model successfully
- only superusers can call delete-model successfully
- only superusers can call delete-provider successfully

Ordinary-user attempts must return an explicit failure instead of success.

### 2. Backend result correctness

Delete endpoints must not always return success.

Required behavior:

- stop returning success on zero-effect delete
- distinguish between:
  - not allowed
  - not found / no matching model

Exact error wording can follow existing API style, but it must not produce a success code for a failed delete.

### 3. Frontend role-aware action handling

Relevant files:

- `web/src/pages/user-setting/setting-model/components/modal-card.tsx`
- `web/src/pages/user-setting/setting-model/hooks.tsx`
- `web/src/hooks/use-llm-request.tsx`
- `web/src/hooks/use-user-setting-request.tsx`

Use current logged-in user role to drive UI behavior.

Expected result:

- superuser can use add/delete entry points
- ordinary user cannot perform add/delete
- ordinary user gets a clear guidance message
- no fake success toast after failed delete

### 4. Add-model flow must also be covered

This ticket is not delete-only.

The following add flows must be checked and restricted consistently:

- provider/API-key add entry if applicable
- model add entry
- local model add/edit modal entry where relevant

If product behavior differs between “add provider config” and “add model”, document that clearly in the completion record. Do not guess silently.

## Implementation Guidance

### Backend

Recommended approach:

- add explicit `current_user.is_superuser` checks to:
  - `/add_llm`
  - `/delete_llm`
  - `/delete_factory`
- return a non-success code plus readable message for ordinary users
- ensure delete result checks affected rows before returning success

### Frontend

Recommended role source:

- `useFetchUserInfo()`

Expected UX:

- add/delete entries are either hidden or disabled for ordinary users
- if ordinary users still reach an add/delete action, show a clear message:
  - add: contact superuser
  - delete: contact superuser
- success toast must only appear on real success
