# 2026-04-04 Chat Retrieval Overall Solution

## 目标

在不大改开发平台配置模型的前提下，优化聊天场景中的知识库检索、知识图谱增强与深度检索体验。

本方案遵循两条原则：

1. 现有前端按钮保留
   - `Thinking`
   - `Use knowledge graph`
   - 现有检索相关配置项

2. 设置界面继续保留开发者自由度
   - 不做激进的配置重构
   - 以“新增能力接入”和“补齐真实语义”为主

## 现状问题

### 1. 检索策略被过度绑定到 prompt 结构

- 当前只要 dialog 绑定知识库，且 prompt 参数中存在 `knowledge`，就会默认进入检索
- 导致很多不该检索的问题也会强制查库

### 2. `Thinking` 按钮实际上在切换检索质量

- 现在 `Thinking` 不只是展示推理
- 它还会把请求切到 `DeepResearcher`
- 这造成用户体感上“不点 Thinking 就不像真正用到了知识库”

### 3. `Use knowledge graph` 不是检索总开关

- 它当前只控制是否额外走 KG retriever
- 不能控制普通知识库检索是否执行

### 4. 图谱结果已进入后端，但前端感知弱

- KG 结果当前被包装成伪 chunk
- 前端引用展示仍以文档 chunk / doc_aggs 为核心
- 用户容易觉得“开了知识图谱但没看到图谱召回”

## 总体思路

核心原则：

- 保留现有交互按钮
- 不把能力删掉
- 不把设置界面做复杂
- 将“能力”和“策略”解耦

即：

- `Thinking` 继续存在，但不再成为高质量知识库回答的唯一入口
- `Use knowledge graph` 继续存在，但明确为“是否允许图谱增强”
- 新增一个轻量但真实的“检索策略层”

## 推荐能力模型

### 一、能力层

保留并明确三种能力：

1. 标准知识库检索
   - 全文 + 向量混合

2. 知识图谱增强
   - 图谱实体、关系、community summary

3. 深度检索
   - query rewrite
   - decomposition
   - iterative retrieval

### 二、策略层

新增一个内部策略概念，但不要求首期在 UI 中大改：

- `retrieval_mode`
  - `auto`
  - `always`
  - `off`

- `deep_research_mode`
  - `auto`
  - `manual`
  - `off`

- `kg_mode`
  - `off`
  - `on`

其中：

- `kg_mode` 可以直接复用现在的 `use_kg`
- `Thinking` 按钮首期仍然作为 `deep_research_mode=manual` 的一次性触发入口

## 前端交互方案

### 1. 保留 `Thinking` 按钮

语义调整为：

- 点击后：明确请求“深度检索模式”
- 不点击时：系统仍可能在 `auto` 下自动升级检索，但不进入完整人工显式 `Thinking` 体验

这样做的好处：

- 保留现有用户习惯
- 保留高级能力显式入口
- 避免“不点按钮就答不好”的问题

### 2. 保留 `Use knowledge graph`

语义明确为：

- 允许在本次 chat 中使用知识图谱增强
- 不等于“强制图谱优先”

### 3. 设置界面不大改

开发者设置页首期只建议增加 1 个轻量选项：

- 检索模式：
  - 自动
  - 始终检索
  - 关闭检索

其余已有项尽量不动。

如果首期想更稳，甚至可以先不暴露新 UI，只在后端内部默认启用 `auto`。

## 后端策略方案

### 1. 标准默认模式改为 `auto`

不再继续沿用“只要挂 KB 就强制检索”的策略。

建议：

- 简单闲聊、润色、翻译、通用推理：不检索
- 常规知识问答：标准检索
- 多约束、多实体、多跳、跨文档问题：自动升级

### 2. 标准检索保持混合检索底座

保留：

- 原始 query -> 向量检索
- 关键词增强 / term-weight / synonym 扩展后的 query -> 全文检索
- 融合后取结果

不建议：

- 退回纯向量
- 退回纯 ES
- 强制只用原 query
- 强制只拆关键词

### 3. 自动升级到深度检索

触发条件建议：

- 问题包含多个实体/条件/年份/组织
- 问题明显要求比较、因果、关系链、跨文档归纳
- 首轮召回不足
- 首轮结果分数偏低
- 标准检索后 evidence 不足

### 4. 图谱只作为增强层

图谱启用时：

- 对简单问题，不必强行插图谱
- 对关系型、多跳型、全局总结型问题，再启用图谱增强

## 图谱展示方案

这是本次很关键的一部分。

### 当前问题

KG 结果当前作为伪 chunk 混进普通引用区，不利于理解。

### 推荐方案

新增独立的 `Graph Evidence` 展示区：

- 实体命中
- 关系命中
- community summary

保留原文档引用区不变：

- 文档 chunk
- 文档来源
- 页码 / 位置 / 图片预览

这样用户就能明确看到：

- 哪部分来自知识库文档
- 哪部分来自知识图谱推理

## 对开发平台的影响

本方案刻意控制改动面：

### 保留自由度

- 系统 prompt 仍可自定义
- 是否启用知识图谱仍可配置
- 是否手工触发深度检索仍可配置
- 现有高级参数仍保留

### 不做的事情

- 不强制删除现有设置项
- 不重做 chat 配置页面
- 不改成“只有一种官方固定工作流”

### 新增的只是更真实的底层语义

- `Thinking` = 显式深度检索入口
- `use_kg` = 图谱增强许可
- `retrieval_mode` = 是否执行知识库检索的真实策略

## 推荐实施顺序

### 第一阶段

- 将默认检索模式切到 `auto`
- 保留 `Thinking`
- 保留 `use_kg`
- 增加自动升级逻辑
- 增加图谱独立展示区

### 第二阶段

- 调整混合检索融合策略
- 评估引入 RRF 或更合理的融合权重
- 优化复杂问题的自动 query rewrite / decomposition

### 第三阶段

- 如果效果确认，再考虑暴露更多高级策略项给开发者
- 如：
  - 深度检索自动阈值
  - 图谱优先策略
  - 检索质量回退策略

## 项目经理结论

最合适的方向不是删能力，而是：

- 保留现有按钮
- 保留开发平台自由度
- 在底层把“检索策略、图谱增强、深度检索”从耦合关系里拆出来

一句话总结：

从“用户必须自己点对按钮才拿到高质量知识库回答”，改成“系统默认更聪明地用检索，用户点击按钮时再获得额外增强能力”。
