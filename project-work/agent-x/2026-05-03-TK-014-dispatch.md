## Incident Background

Remote deployment reported that a configured `qwen3-vl-embedding` model could not be used after being added to RAGFlow.

The current working hypothesis is vector-dimension incompatibility between the embedding model output and the document-engine vector mapping. This is especially likely when the remote document engine is Elasticsearch and the embedding model outputs 2048, 2560, 3072, or 4096 dimensions.

Confirmed reference points:

- Qwen `Qwen3-VL-Embedding-8B` supports output dimensions up to 4096, and the Qwen model list identifies `Qwen3-VL-Embedding-2B` as 2048 dimensions and `Qwen3-VL-Embedding-8B` as 4096 dimensions: https://huggingface.co/Qwen/Qwen3-VL-Embedding-8B
- Qwen `Qwen3-Embedding-8B` supports output dimensions up to 4096; the Qwen3 embedding family also includes 0.6B and 4B variants: https://huggingface.co/Qwen/Qwen3-Embedding-8B
- Google Gemini Embedding 2 supports flexible output dimensions from 128 to 3072, with Matryoshka-style smaller representations: https://deepmind.google/models/gemini/embedding/
- Google Gemini API `gemini-embedding-001` documents flexible output dimensions from 128 to 3072 and recommends 768, 1536, or 3072: https://ai.google.dev/gemini-api/docs/embeddings
- OpenAI `text-embedding-3-large` defaults to 3072 dimensions, while `text-embedding-3-small` defaults to 1536 dimensions: https://platform.openai.com/docs/guides/embeddings

## Current Technical Finding

Code paths inspected before dispatch:

- `rag/svr/task_executor.py`
- `rag/nlp/search.py`
- `api/apps/chunk_app.py`
- `api/db/services/document_service.py`
- `conf/mapping.json`
- `conf/os_mapping.json`
- `rag/utils/es_conn.py`
- `rag/utils/opensearch_conn.py`

Current behavior:

- RAGFlow does not globally hard-code embedding vectors to 1024 dimensions.
- The embedding model is called and the returned vector length is used as `vector_size`.
- Chunk vectors are stored under dimension-specific fields such as `q_1024_vec`, `q_2048_vec`, or `q_4096_vec`.
- Retrieval uses the current query embedding length and searches the matching field name.
- Elasticsearch mapping currently supports only `512`, `768`, `1024`, and `1536` vector fields in `conf/mapping.json`.
- OpenSearch mapping supports more dimensions, including `2048` and `4096`, but misses common dimensions such as `2560` and `3072`.
- Existing indexes may not be repaired by changing the template alone because index creation can be skipped when the index already exists.

## Goal

Make RAGFlow compatible with mainstream embedding vector dimensions across supported document engines, without requiring a code change for every new embedding model.

The fix must support both newly created indexes and already existing remote indexes.

## Required Outcomes

### 1. Define a shared mainstream vector-dimension list

Create a single source of truth or clearly synchronized configuration for supported vector dimensions.

Minimum required dimensions:

```text
128
256
384
512
768
1024
1536
2048
2560
3072
4096
6144
8192
10240
```

The implementation may include additional safe dimensions if justified by common embedding providers.

The `128`, `256`, `768`, `1536`, and `3072` dimensions are especially important for Google Gemini Embedding / Gemini Embedding 2 style Matryoshka output, where operators may reduce vector size to trade accuracy for storage and latency.

### 2. Update Elasticsearch vector mapping

Elasticsearch must support at least the required dimensions for `q_<dim>_vec` fields.

Required behavior:

- `conf/mapping.json` includes dynamic templates for all required vector dimensions.
- Each template maps `*_N_vec` to a dense vector with the matching dimension.
- The mapping remains compatible with the Elasticsearch version used by this deployment.
- The implementation must not silently degrade vectors to text or non-indexed fields.

### 3. Update OpenSearch vector mapping

OpenSearch must support the same required dimensions unless a specific dimension is not technically supported by the deployed OpenSearch version.

Required behavior:

- `conf/os_mapping.json` includes templates for the required dimensions.
- Missing common dimensions such as `2560` and `3072` must be added.
- If OpenSearch has version-specific limits, document them clearly.

### 4. Inspect and document Infinity/OceanBase behavior

Do not assume this is only an Elasticsearch problem.

Required behavior:

- Inspect Infinity vector table creation and dynamic field behavior.
- Inspect OceanBase vector table creation and dimension handling.
- If a backend already supports dynamic dimensions, document the evidence.
- If a backend has fixed limits, add validation or clear operator guidance.

### 5. Handle existing indexes

This is mandatory. New templates alone are not sufficient for remote deployments with existing indexes.

Required behavior:

- Provide a safe way to update existing Elasticsearch/OpenSearch indexes with the missing dynamic templates or vector mappings.
- The mechanism may be an admin utility script, migration function, or documented operator command, but it must be practical for the remote deployment.
- It must be idempotent.
- It must avoid deleting data.
- It must log which index was checked, which dimensions were missing, and which dimensions were added.
- It must warn when a backend cannot update an existing index in place and requires reindexing or knowledge-base reparse.

### 6. Add early validation and better error messaging

When an embedding model returns a dimension unsupported by the current document engine, operators need an actionable error before or during parsing.

Required behavior:

- Detect the returned embedding dimension.
- Check whether the current document engine can store and search `q_<dim>_vec`.
- If unsupported, show a clear message naming the model, dimension, document engine, and required action.
- Do not let the system fail later with a low-level ES/OpenSearch mapping exception if this can be caught earlier.

### 7. Add verification coverage

Required tests or scripts:

- Verify that all required dimensions exist in `conf/mapping.json`.
- Verify that all required dimensions exist in `conf/os_mapping.json`.
- Verify that generated field names match `q_<dim>_vec`.
- Verify idempotent existing-index upgrade logic with a mocked or local document-engine client where possible.
- Include at least one 4096-dimensional scenario because it covers `Qwen3-VL-Embedding-8B` and `Qwen3-Embedding-8B`.

## Initial Remote Guidance

Until this ticket is implemented, the safest mitigation is:

- For Elasticsearch deployments, use an embedding model whose dimension is already supported by the current mapping, such as 1024 or 1536.
- If using `Qwen3-VL-Embedding-8B` or `Qwen3-Embedding-8B`, configure the embedding service to output 1024 dimensions if the service supports custom output dimensions.
- If the model must output 4096 dimensions, create a new knowledge base only after the mapping has been upgraded, then reparse documents so chunks are stored under `q_4096_vec`.
- Existing knowledge bases parsed with a different embedding dimension must be reparsed or rebuilt for reliable vector retrieval.
