# 2026-05-03 TK-013 Dispatch

## Basic Information

- Ticket ID: `TK-013`
- Title: GraphRAG large-scale embedding throughput and fault-tolerance remediation
- Assignee: `agent-B`
- Dispatcher: `agent-x`
- Status: `dispatched`
- Priority: `high`
- Type: `special-remediation`

## Incident Background

Remote runtime reported a GraphRAG generation failure during node embedding:

```text
Get embedding of nodes: 32709/32713
[ERROR][Exception]: Operation timed out after 10800 seconds and 1 attempts.
```

The remote RAGFlow instance uses a privately deployed embedding service rather than a public embedding API. The observed failure is consistent with throughput saturation and long queueing during GraphRAG node/edge vectorization.

This is not only a timeout setting issue. The current GraphRAG path can generate tens of thousands of nodes and relations, and the current embedding conversion path does not have enough backpressure, batching, retry, or partial-progress tolerance for that scale.

## Current Technical Finding

Code paths inspected:

- `rag/graphrag/utils.py`
- `rag/llm/embedding_model.py`
- `rag/svr/task_executor.py`
- `common/connection_utils.py`
- `common/misc_utils.py`

Current behavior:

- `set_graph()` creates one async task per changed node and one async task per changed edge.
- `graph_node_to_chunk()` calls `embd_mdl.encode([ent_name])`, effectively one embedding request per node.
- `graph_edge_to_chunk()` calls `embd_mdl.encode([edge_text])`, effectively one embedding request per edge.
- `chat_limiter` defaults to `MAX_CONCURRENT_CHATS=10` and is shared with chat/model operations.
- `thread_pool_exec()` can use up to `THREAD_POOL_MAX_WORKERS=128`.
- The whole task is capped by `do_handle_task` at `10800` seconds.
- If the embedding phase fails near the end, the task is interrupted after many successful embeddings, but the current graph-chunk conversion does not persist fine-grained node/edge embedding progress.

## Goal

Make GraphRAG generation reliable for large graphs under privately deployed embedding services.

The fix must handle workloads with tens of thousands of nodes/edges without requiring manual babysitting or blind timeout increases.

## Required Outcomes

### 1. Batch node and edge embedding

Replace per-node/per-edge single-item embedding calls with batch embedding.

Required behavior:

- Collect node texts into batches before calling `embd_mdl.encode`.
- Collect edge texts into batches before calling `embd_mdl.encode`.
- Add explicit configuration for GraphRAG embedding batch size.
- Default must be conservative for private deployments.

Suggested environment variables:

```text
GRAPHRAG_EMBED_BATCH_SIZE=16
GRAPHRAG_EMBED_CONCURRENCY=2
```

Do not rely on `MAX_CONCURRENT_CHATS` as the only control knob for GraphRAG embedding.

### 2. Separate GraphRAG embedding concurrency from chat concurrency

Create a dedicated limiter for GraphRAG embedding.

Required behavior:

- Use a dedicated `asyncio.Semaphore` for node/edge embedding.
- Keep `MAX_CONCURRENT_CHATS` for chat/LLM calls.
- Add a lower default concurrency for GraphRAG embedding to protect private embedding services.

### 3. Increase and parameterize task timeout

The current 3-hour task timeout is not sufficient for large private deployments.

Required behavior:

- Make GraphRAG/task timeout configurable by environment variable.
- Keep a sane default, but allow remote deployment to raise it without code changes.
- Timeout increase must not be the only fix.

Suggested environment variables:

```text
RAGFLOW_TASK_TIMEOUT_SECONDS=10800
GRAPHRAG_TASK_TIMEOUT_SECONDS=21600
```

### 4. Add retry with backoff for embedding batches

Embedding calls must retry transient failures.

Required behavior:

- Retry timeout, connection reset, 429/rate-limit, and 5xx-like provider failures where detectable.
- Use bounded retry attempts.
- Use exponential backoff plus jitter.
- Log batch index, batch size, attempt number, and elapsed time.
- Do not retry non-transient input/configuration errors indefinitely.

Suggested environment variables:

```text
GRAPHRAG_EMBED_MAX_RETRIES=3
GRAPHRAG_EMBED_RETRY_BASE_SECONDS=2
GRAPHRAG_EMBED_RETRY_MAX_SECONDS=60
```

### 5. Add ordered internal queue/backpressure for large embedding jobs

Large graph embedding must not enqueue tens of thousands of unbounded tasks at once.

Required behavior:

- Process node/edge batches through an ordered queue or bounded task pool.
- Preserve deterministic output mapping from input node/edge to vector.
- Avoid creating one async task for every node/edge.
- Report progress by completed batches and completed items.

Expected progress examples:

```text
Get embedding of nodes: 32000/32713, batches 2000/2045
Retry node embedding batch 1987/2045 attempt 2/3 after timeout
Get embedding of edges: 18000/22000, batches 1125/1375
```

### 6. Improve GraphRAG fault tolerance for very large graphs

Large graphs make model/provider failures more likely. The GraphRAG run must fail safely and resume cleanly.

Required behavior:

- If an embedding batch fails after all retries, mark the task as interrupted/resumable.
- Preserve the previous durable graph until the new graph is fully ready.
- Do not delete the old graph before all new vector chunks are successfully prepared.
- Record enough failure context for operators to know whether to resume, reduce concurrency, or regenerate.
- Integrate with existing `resume_failed` behavior where practical.

### 7. Add configuration guidance for private embedding deployments

Add documentation under `project-work` or `docs` describing recommended settings for private embedding services.

Required content:

- Small private CPU service recommendations.
- Single-GPU service recommendations.
- Multi-worker or high-throughput service recommendations.
- How to tune batch size, concurrency, retry, and timeout.
- How to recognize service saturation from logs.

## Acceptance Criteria

`agent-x` will not accept this ticket unless all of the following are satisfied:

- GraphRAG embedding no longer creates one unbounded async task per node/edge.
- Node and edge embedding use batch calls.
- GraphRAG embedding has its own concurrency controls.
- Embedding batch retries are implemented with bounded backoff.
- Large jobs report useful batch-level progress.
- A failed batch leaves the task resumable and preserves the previous graph.
- Timeout is configurable without code change.
- Documentation includes remote private embedding tuning guidance.
- The assignee provides test evidence for at least one synthetic large graph or mocked embedding workload.

## Minimum Self-Test Requirements

Before handoff, the assignee must report:

- Unit or script-level test with mocked embedding delay and transient failures.
- Verification that batching reduces request count.
- Verification that retry succeeds after a transient failure.
- Verification that permanent failure reports a resumable/interrupted task.
- Verification that old graph deletion still happens only after vector chunks are prepared.
- A recommended remote environment setting for the reported 32k-node case.

## Non-Goals

- Do not redesign GraphRAG extraction prompts.
- Do not change retrieval ranking behavior unless strictly needed for compatibility.
- Do not solve private embedding service deployment itself; only make RAGFlow a good client of it.
- Do not hide failures by only increasing timeout.

## Initial Recommended Remote Mitigation

Until this ticket is implemented, remote operators should use conservative settings:

```text
MAX_CONCURRENT_CHATS=2
THREAD_POOL_MAX_WORKERS=32
```

If the embedding service supports stable batching and has GPU capacity, increase gradually:

```text
MAX_CONCURRENT_CHATS=3
THREAD_POOL_MAX_WORKERS=64
```

Avoid running multiple large GraphRAG jobs against the same private embedding endpoint at the same time.
