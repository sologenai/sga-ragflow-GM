# 2026-04-29 KB Label Auto Classification Fix

## Background

Remote verification reported that knowledge bases generated by the external real-time chat graph flow were created successfully, but were not automatically grouped under the expected dataset label.

The relevant historical feature is the fixed knowledgebase label system:

- `manual`: 后台创建
- `chat_graph`: 聊天图谱
- `news_sync`: 新闻同步
- `archive_sync`: 档案同步
- empty string: 未标注

This feature is implemented through the `knowledgebase.kb_label` field, not through the document-level tag system.

## Root Cause

The normal logged-in backend endpoint `/kb/create` already forwarded `kb_label` into `KnowledgebaseService.create_with_name(...)`.

However, the API-token dataset endpoint `/api/v1/datasets`, which is the likely entry used by the external real-time graph frontend, validates requests through `CreateDatasetReq`. That schema did not expose `kb_label`, so the fixed business label was not reliably accepted or persisted through this API path.

There was also no backend fallback for the earlier `source=frontend_graph` wording from the first proposal, and no fallback for the historical chat graph dataset name pattern.

## Changes

- Added `kb_label` support to the API-token dataset create/update schema.
- Added `source` as a backward-compatible alias for `kb_label`.
- Normalized `source=frontend_graph` to `kb_label=chat_graph`.
- Added backend fallback recognition for chat graph dataset names matching `^.+_\d{8}_\d{4}(_.*)?$`.
- Added read-time fallback for list/detail responses: unlabeled KBs with chat-graph-style names are returned as `chat_graph`.
- Ensured news-sync-created KBs are saved or corrected as `news_sync`.
- Ensured archive-sync-created KBs are saved or corrected as `archive_sync`.
- Updated SDK dataset endpoint docs to mention `kb_label` and `source`.

## Remote Retest

For external real-time graph creation, the preferred request field is:

```json
{
  "kb_label": "chat_graph"
}
```

The older compatible field is also supported:

```json
{
  "source": "frontend_graph"
}
```

If neither field is sent, the backend now falls back to the historical chat graph naming rule and labels names like:

```text
username_20260429_1530
```

as `chat_graph`.

Existing records created before this fix are also covered at read time when they are still unlabeled and match the same naming rule. Manual labels are not overwritten.
