# 2026-04-04 KB Origin Tagging Proposal

## 问题背景

外部前端项目在与用户聊天过程中，会通过超级管理员 token 在 RAGFlow 后端自动创建知识库，并构建实时知识图谱。

当前这些知识库会与后台人工创建的知识库混在同一个列表中，名称多为“用户名 + 时间”等模式，导致超级管理员很难快速识别：

- 哪些是后台人工维护的正式知识库
- 哪些是外部前端自动创建的临时/过程知识库

## 当前代码现状

- `knowledgebase` 表当前没有“来源”字段。
- 知识库列表接口仅支持按关键字、分页、owner、parser 等常规条件查询。
- 前端知识库总列表页面当前只有 owner 维度筛选，没有来源维度筛选。

## 建议方案

采用“后端正式来源字段 + 前端标签显示 + 列表筛选按钮”方案。

### 1. 后端增加知识库来源字段

建议在 `knowledgebase` 表新增字段，例如：

- `source`

候选枚举值：

- `manual`
- `frontend_graph`
- `system`

默认值建议为：

- `manual`

这样可以保证现有历史数据自动归类为后台人工创建知识库，不影响已有功能。

### 2. 创建知识库时显式写入来源

后台原生创建知识库：

- 默认写入 `manual`

外部前端自动创建知识库：

- 调用创建接口时显式传入 `source=frontend_graph`

系统自动任务创建的知识库（如新闻、档案）：

- 可按需要后续补为 `system`

### 3. 列表接口支持按来源筛选

在知识库列表接口增加可选筛选参数：

- `source`

支持：

- 单值筛选
- 多值筛选

### 4. 前端列表增加来源识别能力

建议同时做两层可视化区分：

- 卡片上增加小标签
- 顶部增加来源筛选按钮

建议按钮选项：

- 全部
- 后台创建
- 前端图谱

### 5. 历史数据处理

第一阶段可不做复杂回填。

策略：

- 历史数据统一按 `manual`
- 新创建的数据按真实来源写入

如果后续确实需要清理历史“前端生成知识库”，再单独做一次回填或批量标记工单。

## 为什么不建议只靠名称规则

如果继续只靠“用户名 + 时间”命名规则区分，会有这些问题：

- 规则脆弱，名称一改就失效
- 无法可靠筛选
- 后续做自动清理、统计、权限隔离都很难
- 无法从接口层稳定区分来源

## 推荐落地顺序

### 第一阶段

- 增加 `knowledgebase.source`
- 创建接口支持传入来源
- 列表接口返回来源
- 列表页面增加标签显示
- 列表页增加来源筛选按钮

### 第二阶段

- 对新闻/档案等系统知识库补 `system`
- 支持按来源批量清理
- 支持按来源权限隔离或独立视图

## 预估改动范围

后端：

- `api/db/db_models.py`
- `api/db/services/knowledgebase_service.py`
- `api/apps/kb_app.py`

前端：

- `web/src/interfaces/database/knowledge.ts`
- `web/src/interfaces/request/knowledge.ts`
- `web/src/hooks/use-knowledge-request.ts`
- `web/src/pages/datasets/index.tsx`
- `web/src/pages/datasets/dataset-card.tsx`

外部前端项目：

- 创建知识库时补充来源参数 `source=frontend_graph`

## 风险判断

- 风险较低
- 属于标准的“加字段 + 扩筛选 + 扩展示”的改动
- 对现有知识库主流程影响很小

## 项目经理建议

这件事值得立为一个单独工单，优先级建议为中高。

工单目标建议定义为：

- “实现知识库来源标记与筛选，区分后台人工创建与前端自动构建知识库”
