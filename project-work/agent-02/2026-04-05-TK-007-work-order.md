## Background

Local runtime verification on the current branch showed that two related account-management problems are still present:

- password changes are still inconsistent across real user paths
- the expected admin-side unlock capability for login-locked users is not present

This ticket intentionally combines both issues because they are part of the same operational support surface for account recovery.

Reference analysis:

- `project-work/agent-x/2026-04-05-password-and-unlock-analysis.md`

## Verified Current Findings

### Password side

The earlier accepted conclusion is no longer sufficient after live verification.

Current verified behavior:

- active login paths still authenticate against `hash(plain password)` after decrypt + base64 decode
- admin create user currently passes base64 password representation into account creation
- admin reset password currently compares/stores base64 representation
- personal-settings password change currently verifies and stores base64 representation
- default superuser initialization still seeds plain `admin`

Result:

- admin-reset passwords can become unusable for login
- personal-settings password change can fail even when the user enters the real current password

### Unlock side

Current verified behavior:

- login failure locking exists in Redis
- keys are:
  - `login_attempts:<email>`
  - `login_lock:<email>`
- admin UI does not expose lock state
- admin API does not expose unlock action
- admin UI does not provide a one-click unlock operation

Result:

- product expectation is not met
- users can be locked by failed logins without a supported admin-side recovery action

## Goals

Repair the account recovery surface so that:

- admin-created users can log in with their assigned password
- admin-reset passwords work for login
- users can change password from personal settings using their real current password
- locked users receive a clear "contact administrator" style message
- admins can see lock state and unlock the user from the management UI

## Mandatory Implementation Principles

- use the smallest correct fix
- align all affected password write/check paths to the real active login convention
- do not redesign the entire authentication system
- do not change the frontend RSA/base64 transport protocol
- do not silently skip the unlock half of this ticket
- add regression coverage for both password correctness and unlock behavior

## Required Changes

### 1. Correct password handling to the active login convention

Primary files to inspect:

- `admin/server/auth.py`
- `admin/server/services.py`
- `api/apps/user_app.py`
- `api/db/joint_services/user_account_service.py`
- `api/db/services/user_service.py`
- `api/utils/crypt.py`

Required outcome:

- after decrypt + base64 decode, plain password is the value used for hash comparison/storage wherever the live login chain expects it

This includes at minimum:

- admin create user
- admin reset user password
- personal-settings password change
- default superuser initialization review

### 2. Restore admin unlock capability

Primary files to inspect:

- `admin/server/routes.py`
- `admin/server/services.py`
- `web/src/pages/admin/users.tsx`
- `web/src/services/admin-service.ts`
- `api/utils/web_utils.py`

Required outcome:

- admin can see whether a user is login-locked
- admin can trigger unlock from the admin UI
- unlock clears both Redis login-lock keys for that user

Minimum backend behavior:

- add a user unlock action that deletes:
  - `login_attempts:<normalized_email>`
  - `login_lock:<normalized_email>`

### 3. Improve lock message

Primary files to inspect:

- `api/apps/user_app.py`
- `admin/server/auth.py`

Required outcome:

- when login is denied because the account is locked, the response message should guide the user to contact an administrator for unlock

### 4. Add regression coverage

You must add focused regression coverage proving at minimum:

- admin-created user password is stored compatibly with login
- admin-reset password is stored compatibly with login
- personal-settings password change uses the correct current-password representation
- unlock clears Redis lock state
