## Background

Customer feedback indicates that password modification is failing in the following business paths:

- administrator resets a user password from the admin UI
- user later changes password from personal settings

This issue is not treated as an ad-hoc fix. It must be written down as a traceable remediation ticket because it has happened before.

Project-manager analysis has already confirmed a code-level root cause:

- the admin UI password reset uses the independent admin server, not the main `user_app` password-setting route
- the bug is in `admin/server/services.py` within `UserMgr.update_user_password`
- the current implementation mixes plain-text password and base64 password representations
- this can corrupt password storage format
- once a password record is corrupted by the admin reset flow, the personal-settings password change flow can also fail later

Reference analysis document:

- `project-work/agent-x/2026-04-04-password-change-bug-analysis.md`

## Root Cause To Fix

Current problematic logic is in:

- `admin/server/services.py`

Current issue:

1. `decrypt(new_password)` returns the base64-encoded password string used by the login chain
2. the code decodes it to plain text for strength validation
3. but the code currently:
   - compares hash with plain text
   - stores plain text through `UserService.update_user_password`

System convention elsewhere is different:

- password hashing and verification consistently use the base64 password representation after RSA decrypt

This mismatch is the primary defect to repair.

## Goals

Repair the password reset chain so that:

- admin password reset writes password in the correct format
- accounts reset by admin can log in with the new password
- those accounts can later modify password again from personal settings
- recurrence risk is reduced by regression coverage
