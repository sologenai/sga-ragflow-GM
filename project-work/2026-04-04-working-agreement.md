# 2026-04-04 Working Agreement

## 协作原则

- 后续所有修改点都通过“工单”推进。
- 一个工单对应一个独立线程，避免多个需求混线。
- 项目经理只负责需求澄清、分派、跟踪、测试验收、结案。
- 功能实现由被分派的执行 agent 负责。

## Agent 命名规则

- 涉及基础设施建设、机制搭建、平台治理类任务，执行 agent 按 `agent-a`、`agent-b`、`agent-c` 形式命名。
- 涉及专项整改、具体业务工单、定向功能修复类任务，执行 agent 按 `agent-001`、`agent-002`、`agent-003` 形式命名。
- 项目经理角色保留为 `agent-x`。
- 每个 agent 在 `project-work/` 下拥有独立目录。

## 目录规则

- 项目级状态记录：放在 `project-work/` 根目录。
- 项目级文档统一使用带日期的 `.md` 文件。
- agent 执行记录：放在各自目录内，文件同样使用日期命名。

## 工单最少应包含的信息

- 工单编号或标题
- 目标
- 范围
- 不在范围内的内容
- 涉及模块或文件
- 风险点
- 验收标准
- 自测要求

## 线程工作方式

- 你在新线程中提交具体工单。
- 我以项目经理身份判断是否受理、如何拆分、分配给哪个 agent。
- 执行 agent 完成后，由我统一进行测试验收。
- 验收通过后，在 `project-work/` 中留下结论记录。

## 验收原则

- 先看是否满足工单目标。
- 再看是否超范围改动。
- 再看是否有回归风险。
- 最后看测试、自测、日志或截图证据是否足够。

## 当前约定

- 本线程用于项目管理机制与全局状态维护。
- 后续具体修改，请开新线程按工单推进。
