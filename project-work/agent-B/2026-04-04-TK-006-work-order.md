## Background

The current chat product has three coupled problems:

1. knowledge-base retrieval is effectively too easy to trigger
2. the `Thinking` button currently behaves like a hidden quality switch for stronger retrieval
3. `use_kg` may run knowledge-graph retrieval, but users often cannot clearly see graph evidence in the chat UI

Project-manager research has concluded that the first phase should not be a full GraphRAG redesign.
Instead, it should:

- keep existing frontend controls
- preserve developer freedom in the chat settings page
- improve default retrieval behavior
- make graph evidence visible without breaking the current document-reference model

## Phase-1 Goal

Deliver a controlled first-phase improvement with the following product principles:

1. keep the existing `Thinking` button
2. keep the existing `Use knowledge graph` capability
3. do not heavily redesign the settings page
4. improve retrieval strategy under the hood
5. make graph evidence visible to the user

## Required Outcomes

### 1. Preserve current controls

The following must remain available:

- frontend `Thinking` button
- frontend `Use knowledge graph` switch
- existing developer-facing chat settings

This ticket is not allowed to remove or hide those capabilities.

### 2. Introduce phase-1 retrieval strategy behavior

The system should move away from “force retrieval for almost everything” toward a smarter default.

Phase-1 target:

- standard chat no longer depends on the user clicking `Thinking` to get reasonable knowledge-base behavior
- retrieval behavior should support an automatic strategy path
- explicit `Thinking` should still trigger stronger/deeper retrieval behavior

Implementation can be incremental, but the behavior must clearly separate:

- normal retrieval path
- deep retrieval path
- optional graph enhancement

### 3. Keep the current mixed retrieval foundation

Do not regress the current mixed retrieval base.

The first phase should continue to work with:

- full-text retrieval
- vector retrieval
- existing rerank / retrieval enhancement chain where applicable

This ticket is not for replacing the retrieval engine with a new stack.

### 4. Make graph evidence visible in chat

This is a required first-phase outcome.

Current KG results are not clearly visible because they are injected as pseudo chunks and do not naturally fit the current document-reference display model.

Phase-1 expectation:

- users can clearly see when graph evidence participated
- graph evidence should not rely only on hidden pseudo-chunk behavior
- the solution should coexist with current document citations

The implementation does not need to be final-form GraphRAG UX, but it must make the graph contribution visible and understandable.

### 5. Avoid heavy settings-page redesign

The settings page is a developer platform and should keep flexibility.

Allowed:

- a small additional retrieval-strategy option if truly necessary
- internal default behavior improvements without forcing a large new UI

Not allowed:

- large configuration-page redesign
- removing existing prompt freedom
- replacing the developer platform with a rigid fixed workflow

## Suggested Execution Direction

Recommended first-phase direction:

1. keep `Thinking` as an explicit deep-retrieval trigger
2. keep `use_kg` as the graph-enhancement permission
3. add or internalize an automatic retrieval strategy for the normal chat path
4. add a distinct graph-evidence presentation area in chat

This direction is preferred over:

- forcing all chat through retrieval
- forcing all graph evidence into the existing document citation area
- tying answer quality only to the `Thinking` button
