- Title: GraphRAG checkpoint resume completion for timeout/quota failures

## Background

TK-010 restored the visible GraphRAG `Resume / Regenerate` path and the Redis task-monitor compatibility surface. A follow-up source review found that the execution path still did not fully use the checkpoint APIs:

- `GraphRAGTaskMonitor.init_doc_progress(...)`
- `GraphRAGTaskMonitor.update_doc_status(...)`
- `GraphRAGTaskMonitor.get_merged_doc_ids(...)`
- `GraphRAGTaskMonitor.get_resume_from_task_id(...)`

The practical risk was that a model timeout, quota exhaustion, or model-provider error could leave users with a partial graph but no reliable document-level resume state.

## Scope

Implement complete GraphRAG checkpoint resume behavior for the dataset graph generation path:

1. Split GraphRAG run semantics into three modes:
   - `resume_failed`: interrupted-task checkpoint resume
   - `incremental`: process newly added files without deleting the graph
   - `regenerate`: delete the old graph and rebuild from scratch
2. Persist per-document progress for each GraphRAG task.
3. Mark documents as `extracting`, `merged`, `failed`, or `skipped`.
4. Keep partial failures resumable by setting task progress to `-1` when any document fails.
5. On resume, skip documents already merged in the previous task or already durable in the graph.
6. Avoid running resolution/community post-processing while some source documents failed.
7. Support post-processing resume when the previous failure happened after all documents were merged.
8. Align token SDK GraphRAG run/trace endpoints with the UI resume semantics.
9. Reduce graph-loss risk by generating graph chunks before deleting the old graph records.
10. Make the UI safer:
    - incremental update is only enabled when new files are detected
    - regenerate shows a confirmation warning that the old graph will be deleted
11. Ensure manual cancellation preserves the previous GraphRAG task id and does not delete the existing graph.

## Acceptance Criteria

- A failed GraphRAG task with some successfully merged documents can be resumed without regenerating those merged documents.
- Timeout/quota failures are visible as task failure, not silent success.
- `trace_graphrag` can expose document progress through `doc_summary`.
- Clicking `Resume interrupted run` after fixing the model/quota issue processes only unfinished documents where possible.
- Clicking `Resume interrupted run` after manual cancellation can still find the previous failed task.
- Clicking `Incremental update` is disabled unless the graph summary detects ungraphed documents.
- Clicking `Regenerate` still clears old graph data and performs a full rebuild.
- Clicking `Regenerate` requires an explicit confirmation.
- The token SDK path accepts `resume: true` and behaves consistently with the UI path.

## Out Of Scope

- Full graph transaction support in Elasticsearch/Infinity.
- New frontend design changes.
- Provider-specific retry/backoff policy.
