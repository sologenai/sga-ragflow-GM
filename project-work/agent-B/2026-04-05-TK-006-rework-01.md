## Rework Background

`TK-006` was previously accepted as a phase-1 delivery, but live verification on the local Docker environment exposed two runtime-quality failures and one UX discoverability gap:

1. `Thinking` can appear stuck after the initial retrieval lines
2. knowledge-graph evidence is not reliably visible when `Thinking` is off
3. even when graph evidence exists, users do not naturally find it in the current chat reference experience, and visibility is much better when community-summary evidence exists than when only entities/relations are available

This rework keeps the original phase-1 boundaries, but requires those runtime gaps to be corrected before final closure.

Reference review:

- `project-work/agent-x/2026-04-05-TK-006-review-round-1.md`

## Rework Scope

### 1. Fix the knowledge-graph evidence retrieval bug

This is required.

The current graph-retrieval path is producing runtime warnings related to async vector retrieval and may return empty entity/relation evidence.

Expected outcome:

- the KG retrieval path no longer triggers `coroutine ... was never awaited`
- graph evidence is populated correctly when graph retrieval should contribute
- non-thinking + `Use knowledge graph` flow can actually return graph evidence

### 2. Improve `Thinking` progress continuity

This is required.

The current deep-research experience looks frozen after the first visible retrieval lines.

Expected outcome:

- the user sees continued progress during deep retrieval
- long-running deep retrieval no longer looks like a dead UI
- completion / failure paths do not silently leave the frontend hanging in a thinking state

This may involve:

- stronger stage updates
- periodic heartbeat/progress messages
- guaranteed completion/end markers on all code paths

### 3. Make graph evidence easier to notice in chat

This is required.

Phase-1 already added a graph-evidence panel, and live verification confirms that it becomes visible when useful graph evidence such as community summaries exists. The remaining gap is that users still look at the conventional reference area first, and graph participation becomes much less obvious when only entities/relations are available.

Expected outcome:

- graph evidence is visually obvious enough in the assistant message area
- the user can clearly tell when graph retrieval participated
- community-summary evidence should remain the most prominent graph-evidence presentation when available
- when community-summary evidence is not available, entities/relations should still have a clear fallback presentation or the UI should explicitly tell the user that graph retrieval ran but no displayable community summary was produced
- the solution remains within phase-1 scope and does not require a large chat UI redesign

Clarification:

- this rework does not require forcing graph evidence into the traditional document citation list
- the goal is reliable graph-participation visibility, not pretending graph evidence is a normal document chunk
