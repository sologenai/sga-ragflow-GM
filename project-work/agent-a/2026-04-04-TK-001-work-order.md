# 2026-04-04 TK-001 Work Order

## 基本信息

- 工单编号：`TK-001`
- 工单标题：知识库列表按名称规则区分聊天图谱知识库与后台知识库
- 指派对象：`agent-a`

- 优先级：`中高`

## 背景

当前存在一个独立的聊天应用前端。该应用会在与用户聊天过程中，通过超级管理员 token 调用本项目 API 自动创建知识库，并基于这些知识库构建知识图谱。

这些自动生成的知识库会出现在 RAGFlow 的知识库列表中，并与后台人工创建的正式知识库混在一起。由于名称多为“用户名 + 日期 + 时间 + 后缀”形式，超级管理员查看和管理正式知识库时不够方便。

本工单要求采用“最小改动方案”解决该问题。

## 目标

在 RAGFlow 的知识库浏览界面中，通过知识库名称规则，对知识库进行前端本地识别与分组显示，让管理员能够快速区分：

- 后台知识库
- 聊天图谱知识库

## 强制实现原则

- 不改数据库结构
- 不改后端接口
- 不改知识库创建逻辑
- 不改外部聊天应用
- 不引入新的跨项目依赖

## 建议实现方式

### 1. 只改知识库列表页

目标页面：

- `web/src/pages/datasets/index.tsx`
- 相关 hook 与卡片组件

### 2. 通过名称规则判断“聊天图谱知识库”

先将规则封装为独立的判断函数，避免把正则直接散落在页面组件里。

建议规则起点：

- 名称满足类似 `用户名_YYYYMMDD_HHMM_后缀`
- 推荐初版正则：`^.+_\\d{8}_\\d{4}(_.*)?$`

要求：

- 规则必须集中在一个独立函数中，方便后续调整
- 代码里加一行简短注释，说明这是“外部聊天应用自动创建知识库的最小识别规则”

### 3. 页面上增加两类视图切换

在知识库总列表页增加一个简单切换区，推荐以下任一实现：

- 两个按钮
- 两个标签页
- 三段切换：`全部 / 后台知识库 / 聊天图谱知识库`

推荐采用三段切换，这样排查时更方便。

### 4. 数据处理策略

由于本工单不改后端接口，列表区分需要在前端完成。

要求：

- 先拿到知识库列表
- 再按名称规则做本地分类
- 再按当前视图显示对应数据

注意：

- 现有分页是服务端分页
- 若直接在当前页数据上做分类，会导致切换视图后“当前页结果不完整”

最小改动建议：

- 在该页面场景下，优先一次性拉取较大的 `page_size`（例如 500 或 1000，按代码实际情况决定）
- 在前端完成分类与本地分页

如果最终仍保留服务端分页，必须在完成说明里明确写出局限，不可隐瞒。

### 5. 卡片显示小标签

在知识库卡片上增加一个轻量标签，便于肉眼识别：

- 后台知识库：`后台`
- 聊天图谱知识库：`图谱`

要求：

- 样式轻量，不破坏现有卡片布局
- 不影响点击进入知识库详情
