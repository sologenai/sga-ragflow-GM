# ragflow_retrieval 参数配置说明

本文用于说明在另外一个环境里配置 `ragflow_retrieval` 工具时，各个参数应该怎么填、为什么这么填，以及常见错误怎么避免。

这份说明适用于两类场景：

1. 在工作流、智能体编排页面里手工配置 `ragflow_retrieval`。
2. 在 MCP 客户端、工具节点或中间平台里，对接 RAGFlow 检索能力。

## 一、先说结论：一套可直接照填的推荐配置

如果你只是想先把功能跑通，优先使用下面这套配置：

```json
{
  "dataset_ids": ["你的知识库ID"],
  "document_ids": [],
  "question": "{{用户问题}}",
  "page": 1,
  "page_size": 10,
  "similarity_threshold": 0.2,
  "vector_similarity_weight": 0.3,
  "keyword": false,
  "top_k": 1024,
  "rerank_id": "",
  "force_refresh": false
}
```

如果界面里是“变量引用”方式，可以按下面的思路配置：

- `dataset_ids`：绑定一个数组变量，或者直接填固定知识库 ID 列表。
- `document_ids`：没有特殊需要时留空数组 `[]`。
- `question`：绑定用户当前提问内容。
- 其他参数：先按上面的固定默认值填写。

## 二、每个参数怎么填

### 1. `dataset_ids`

作用：

指定要检索哪些知识库。

推荐填写：

```json
["知识库ID1", "知识库ID2"]
```

为什么这么填：

- 这是检索范围的核心参数。
- 传入后，系统只会在你指定的知识库里查。
- 如果不传，MCP 侧会尝试自动取当前 API Key 可见的全部知识库再一起查，虽然也能工作，但范围太大时更容易带来噪声和额外开销。

填写建议：

- 最稳妥的方式是明确填写目标知识库的真实 ID。
- 如果这个智能体只服务一个业务域，通常只配一个知识库 ID。
- 如果是跨专题问答，可以配多个知识库 ID。

注意事项：

- 这里要填的是 ID，不是知识库名称。
- 这里的数据类型必须是数组，哪怕只有一个知识库，也要写成 `["kb_xxx"]`，不能写成 `"kb_xxx"`。

### 2. `document_ids`

作用：

把检索范围进一步缩小到指定文档。

推荐填写：

```json
[]
```

为什么这么填：

- 大多数场景下，我们希望在整个知识库内检索，而不是只查某几个固定文件。
- 留空数组表示不限制具体文档，让系统自行在知识库里召回最相关内容。

什么时候要填：

- 你明确知道只能查某一批文档。
- 你做的是“指定文档问答”而不是“知识库问答”。

注意事项：

- 同样必须是数组。
- 如果传了错误的文档 ID，结果可能直接变少甚至为空。

### 3. `question`

作用：

用户本次要检索的问题文本。

推荐填写：

```text
{{用户问题}}
```

为什么这么填：

- 这是检索输入的主体，没有它就不知道要查什么。
- 通常应该绑定聊天输入、工作流上游节点输出，或标准问题变量。

填写建议：

- 直接传用户原始问题即可。
- 不要先把问题改写得太复杂，先保证原问题进入检索。
- 如果上游已经做了 query rewrite，就绑定改写后的问题变量。

注意事项：

- 不能为空。
- 如果变量没传进来，最终会导致空检索或异常结果。

### 4. `page`

作用：

结果页码。

推荐填写：

```json
1
```

为什么这么填：

- 大部分问答场景只取第一页结果。
- 检索增强回答通常不需要翻页，第一页就足够。

什么时候改：

- 只有在你做“列表型检索展示”时，才需要第 2 页、第 3 页。

### 5. `page_size`

作用：

每页返回多少条结果。

推荐填写：

```json
10
```

为什么这么填：

- MCP 工具层默认按 `10` 处理，适合作为通用起点。
- 结果太少，可能召回不够。
- 结果太多，会增加上下文长度、响应成本和噪声。

实现细节说明：

- 工具定义里推荐默认值是 `10`。
- 后端检索接口自身默认值是 `30`。
- 在实际对接时，建议显式填写 `10`，不要依赖隐式默认值，这样不同环境更一致。

什么时候改：

- 问题复杂、知识点分散时，可以试 `15` 或 `20`。
- 如果模型上下文紧张，或者回答太散，可以降到 `5`。

### 6. `similarity_threshold`

作用：

相似度阈值，用来控制“多相似才算命中”。

推荐填写：

```json
0.2
```

为什么这么填：

- 这是当前接口默认值，也是比较稳妥的起点。
- 阈值太高，会漏掉很多本来有价值但表达不完全一致的片段。
- 阈值太低，会召回更多噪声。

调参建议：

- 召回太少：可以降到 `0.1` 或 `0.15`。
- 召回太杂：可以升到 `0.25` 或 `0.3`。

### 7. `vector_similarity_weight`

作用：

向量相似度权重，用来平衡“语义相似”和“词项匹配”。

推荐填写：

```json
0.3
```

为什么这么填：

- 这是当前接口默认值。
- 这个值越高，越偏向语义相似。
- 这个值越低，越偏向关键词/词项匹配。

怎么理解：

- 如果你的问题和知识库文本表达差异较大，但语义接近，适当提高会更有帮助。
- 如果你的业务里专业名词、制度编号、固定术语很多，保持 `0.3` 往往更稳。

调参建议：

- 术语型、制度型、编号型知识库：优先保持 `0.3`。
- 自然语言表达很多、同义改写很多：可以尝试 `0.4` 到 `0.5`。

### 8. `keyword`

作用：

是否启用关键词增强。

推荐填写：

```json
false
```

为什么这么填：

- 当前默认关闭更稳。
- 开启后，系统会尝试从问题中抽取额外关键词，适合某些召回不足场景，但也可能引入偏词或误扩展。

什么时候可以开：

- 用户问题很短。
- 业务词比较固定。
- 你观察到纯语义召回总是差一点，但关键词检索能明显补强。

什么时候先别开：

- 你还在做第一轮环境联调。
- 你优先追求结果稳定、可控。

### 9. `top_k`

作用：

底层召回阶段允许保留的候选上限。

推荐填写：

```json
1024
```

为什么这么填：

- 这是接口当前默认值。
- 它不是最终返回给页面的条数，而是检索链路里候选集合的上限控制。
- 保持默认值可以避免在复杂知识库里过早截断候选。

填写建议：

- 没有特别性能压力时，先不要改。
- 如果环境资源很紧张，再考虑往下调。

### 10. `rerank_id`

作用：

指定重排模型。

推荐填写：

```json
""
```

为什么这么填：

- 目标环境如果没有明确可用的 rerank 模型，留空最稳。
- 填了不存在、未授权、已删除的模型 ID，最容易直接报模型错误。

什么时候要填：

- 只有在目标环境已经确认部署并授权了某个 rerank 模型时才填。
- 并且要填真实可用的模型标识，不要照搬别的环境里的旧值。

这是一个高风险项：

- 很多“明明删掉了模型却还报模型错”的问题，本质上就是旧配置、旧 DSL、旧会话里仍然残留了模型引用。
- 所以跨环境复制配置时，`rerank_id` 是最需要重新核对的字段之一。

### 11. `force_refresh`

作用：

是否强制刷新 MCP 侧缓存的知识库/文档元数据。

推荐填写：

```json
false
```

为什么这么填：

- 这个参数影响的是 MCP 侧元数据缓存，不是向量库本身的重新建索引。
- 默认关闭可以减少额外开销。
- 大多数正常对话场景不需要每次都强制刷新。

什么时候改成 `true`：

- 你刚新增了知识库或文档，工具节点还看不到。
- 你怀疑当前环境拿到的是旧的知识库/文档清单。
- 你明确在做联调排障，而不是日常查询。

不要误解：

- 它不能替代知识库解析、索引构建。
- 它也不会自动修复“模型不存在”这类问题。

## 三、推荐填写模板

### 场景 A：单知识库通用问答

```json
{
  "dataset_ids": ["kb_001"],
  "document_ids": [],
  "question": "{{query}}",
  "page": 1,
  "page_size": 10,
  "similarity_threshold": 0.2,
  "vector_similarity_weight": 0.3,
  "keyword": false,
  "top_k": 1024,
  "rerank_id": "",
  "force_refresh": false
}
```

适合：

- 一个智能体对应一个知识库。
- 先求稳定跑通。

### 场景 B：多个知识库联合检索

```json
{
  "dataset_ids": ["kb_policy", "kb_notice", "kb_process"],
  "document_ids": [],
  "question": "{{query}}",
  "page": 1,
  "page_size": 10,
  "similarity_threshold": 0.2,
  "vector_similarity_weight": 0.3,
  "keyword": false,
  "top_k": 1024,
  "rerank_id": "",
  "force_refresh": false
}
```

适合：

- 回答要覆盖多个主题资料库。
- 但前提是这些知识库都属于同一业务域，不然容易召回互相干扰。

### 场景 C：刚联调、怀疑缓存没刷新

```json
{
  "dataset_ids": ["kb_001"],
  "document_ids": [],
  "question": "{{query}}",
  "page": 1,
  "page_size": 10,
  "similarity_threshold": 0.2,
  "vector_similarity_weight": 0.3,
  "keyword": false,
  "top_k": 1024,
  "rerank_id": "",
  "force_refresh": true
}
```

适合：

- 刚加完知识库或文档，想确认工具层是不是还拿着旧缓存。

联调完成后建议改回：

```json
false
```

## 四、为什么建议先按这套值走

这套推荐值的原则是“先稳定、再优化”。

核心思路如下：

1. 先显式指定 `dataset_ids`，避免全库乱搜。
2. 先不限制 `document_ids`，避免人为把范围卡死。
3. `question` 直接绑定真实用户问题，避免中间层误改写。
4. `page=1`、`page_size=10`，控制结果规模，方便模型消费。
5. `similarity_threshold=0.2`、`vector_similarity_weight=0.3`，与现有实现默认值保持一致，先用系统稳态参数。
6. `keyword=false`，先排除关键词扩展带来的不确定性。
7. `rerank_id` 留空，先避开跨环境最容易出问题的模型依赖。
8. `force_refresh=false`，只在排障时临时打开。

## 五、最常见的配置错误

### 1. 把知识库名称当成 `dataset_ids`

错误示例：

```json
["综合行政事务"]
```

正确做法：

```json
["kb_xxxxxxxxx"]
```

原因：

接口要的是 ID，不是展示名称。

### 2. `dataset_ids` 写成字符串而不是数组

错误示例：

```json
"kb_xxxxxxxxx"
```

正确做法：

```json
["kb_xxxxxxxxx"]
```

### 3. `rerank_id` 填了别的环境里的旧模型

现象：

- 报未授权。
- 报模型不存在。
- 报模型已删除但仍被引用。

原因：

- 当前环境没有这个模型。
- 旧智能体、旧会话、旧 DSL 里残留了模型引用。

### 4. `question` 没绑定到真实用户输入

现象：

- 一直检索不到。
- 每次结果都很奇怪。
- 工具虽然执行了，但内容毫不相关。

原因：

- 实际上传入的是空值、固定文案，或者错误变量。

### 5. 把 `force_refresh` 当成“重建知识库”

原因：

- 它只是刷新 MCP 侧元数据缓存。
- 它不是重新解析文档，也不是重建向量索引。

## 六、排障建议

如果在新环境里配置完还是不对，建议按下面顺序排查：

1. 先确认 `dataset_ids` 是真实 ID，不是名称。
2. 再确认 `question` 变量传值正常。
3. 再看 `rerank_id` 是否留空，或者是否填了当前环境真实可用模型。
4. 如果刚新增过知识库或文档，再临时把 `force_refresh` 改成 `true` 测一次。
5. 如果召回太少，再调 `similarity_threshold`。
6. 如果召回方向不准，再观察是否需要开启 `keyword` 或调整 `vector_similarity_weight`。

## 七、实施建议

对于新环境上线，建议按下面顺序落地：

1. 先只配一个知识库做通路验证。
2. 使用本文推荐默认值先跑通。
3. 确认能稳定召回后，再逐步增加更多知识库。
4. 最后再考虑 `keyword`、`rerank_id` 这类增强项。

这样做的原因很直接：

- 先把链路跑通，比一开始就堆很多增强参数更重要。
- 一旦同时引入多个变量，出问题时很难定位到底是哪一项配置导致的。

## 八、补充说明

从当前仓库实现来看，需要特别记住两点：

1. `page_size` 在不同层的默认值并不完全一致，所以生产配置里最好显式填写，不要依赖“默认”。
2. `rerank_id` 和对话模型一样，都是跨环境最容易失效的外部依赖字段，迁移或复制配置时必须重新确认。
